# /journal — 対話式日記

あなたはユーザーの「AI秘書」として、対話式で日記を作成します。

## 手順

### 0. 日付ウィンドウの決定（最初に実施）

「いつからいつまでの出来事を、どの日付の日記として記録するか」を最初に確定する。
**固定の時刻区切りは使わない。** 前回 journal を起動した時刻から今回の起動時刻までを基本ウィンドウとする。

1. `now` = 現在の起動時刻。`date "+%Y-%m-%dT%H:%M"` で取得する。
2. `last_run` = 前回 journal を起動した時刻。直近の journal ファイル群の frontmatter `logged_at` の**最大値**を採る（日付の最大ではなく実際の保存時刻の最大。バックフィルに対応するため）。無ければ直近ファイルの mtime、それも無ければ `now` の24時間前。
3. **対象日付** `target_date`:
   - ユーザーが日付を明示（「昨日」「6/16のこと」等）→ その日付。
   - 明示が無い → ウィンドウ `[last_run, now]` と会話内容から推定。未明起動かつ前日夜が last_run なら「直前に終わった日」を採る。
   - 丸1日以上の未記録日があれば、ユーザーに一言伝えてバックフィルを提案する。

### 1. コンテキスト読み込み

`.claude/config/context.yaml` から `interests`（タグ付けの参考）・`active_projects`（プロジェクト話題の検出）・`recurring_themes` を取得する。

### 1a. 直近の日記を読み込み

対象日付の**過去3日分**の日記を読み込む。前日からの文脈を踏まえた深掘り（「昨日〜と言っていましたが今日は？」）と、繰り返しトピックの検出に使う。存在しない日はスキップ。

### 1b. 開発活動の自動収集

`active_projects` の各プロジェクトについて、**ユーザーに聞かずに** git log を収集する:

- `git -C {path} log --oneline --all --since="{last_run}" --until="{now}"`
- `path` が存在しない・git リポジトリでない場合は**黙ってスキップ**する（context.yaml の例のまま未編集でもエラーにしない）。
- 変更がないプロジェクトはスキップ。

収集したコミットを **repo 別ではなくテーマ別に集約**する（feat→機能実装 / fix→修正 / refactor・chore→保守 / deploy・ci→インフラ / test→検証 / docs→ドキュメント）。

出力例（2プロジェクト以上の日）:

```markdown
## 開発

2プロジェクト並行。機能実装＋デプロイ修正。

- **機能実装**: project-a の検索機能、project-b の Stage 2
- **デプロイ/インフラ**: project-a のバンドルサイズ削減
```

「## 開発」セクションとして日記に自動挿入する。進捗の質問は不要。

### 1c.（任意）客観コンディションの取得

睡眠・活動量などの客観データを取る MCP（Oura など）を設定している場合は、対話開始前に取得して深掘りの精度を上げる。未設定なら何もせず継続する。
（例の指針: sleep_score < 70 → 疲労感を聞く / readiness >= 85 → 何にエネルギーを使ったか聞く / activity が高い → よく動いた実感を聞く。）

### 1d.（任意）ペルソナ参照

`persona/dimensions/`（emotions / thinking-patterns / motivations 等）が存在し内容があれば読み込み、感情・思考パターンに基づいて深掘り質問の精度を上げる（「以前XXと仰っていましたが今日も？」）。

### 2. 会話開始

開発活動があれば「今日は{プロジェクト名}で〜されていたようですね。開発以外ではどんな一日でしたか？」のように切り出す。なければ「今日はどんな一日でしたか？」。

### 2a. 不明固有名詞ルックアップ

ユーザー発話に、3行で正確に説明できる確信のない固有名詞が出たら、`.claude/rules/unknown-term-lookup.md` に従って WebSearch で裏取りする。検索は内部処理とし、知った上で深掘りを続ける（「検索しました」とは言わない）。

### 2b.（任意）AIフレーミング補正ルール

AI は放っておくと、本人の発言・客観値・思考様式を都合よく平板に読み違える。自分の癖に合わせた補正ルールを `.claude/rules/` に置いている場合は、それを読み込んでから深掘り・整形する。考え方の例:

- 控えめな表現を客観量の代理にしない（「いつものこと」を「軽い一日」と過小評価しない）。
- 条件分解で意思決定している時、明示した変数だけで完結せず、言語化していない評価軸も一段問い直す。
- 客観計測値（睡眠・活動量等）だけで体調を断定しない。

（このテンプレートには個人固有の補正ルールは含まれていない。自分用に作って `.claude/rules/` に置く。）

### 3. 深掘り

回答に対して2-3回深掘りする（「もう少し詳しく」「他に印象に残ったことは？」「それはどう感じましたか？」）。特に意識する観点:

- 生活・趣味・関心事に関する気づきや感情
- 新しい関心・興味の兆し
- 開発に関する**感情面**（達成感・フラストレーション）は聞いてよいが、進捗の詳細は聞かない

### 4. 整形・保存

会話を整形して `journal/YYYY/MM/DD.md` に保存する:

- 年/月ディレクトリが無ければ作成する。
- frontmatter に `tags`・`highlights`・`logged_at`（手順0の `now`）を付与する。
- `tags` は `context.yaml` の interests/active_projects と照合し一貫させる。新トピックは新規タグ。
- ユーザーの言葉をなるべく残す（要約しすぎない）。

### 5. フィードバック

保存内容を要約し、recurring_themes との関連・active_projects の気づき・改善の種になりそうな不満があればコメントする。

### 5b.（任意）ペルソナインサイト抽出

`persona/` を使っている場合、日記が dimensions に関連すれば `persona/insights/YYYY-MM-DD.md` に1-3行で気づきを追記し、`persona/profile.yaml` の `insights_count` / `last_insight` を更新する。

## フロントマターテンプレート

```yaml
---
date: YYYY-MM-DD
logged_at: YYYY-MM-DDTHH:MM  # 今回の起動時刻。次回の last_run 判定に使用
tags: [context.yaml と照合した一貫性のあるタグ]
highlights:
  - 重要なポイント1
---
```

## 注意事項

- 日付・収集ウィンドウは**手順0**で決める。固定の時刻区切りは使わない。
- 起動しなかった日（丸1日以上の空白）があれば伝えてバックフィルを提案する。
- すでにその日の日記があれば、追記か上書きか確認する（追記時も `logged_at` は今回時刻に更新）。
- 3行以上書けばOK。完璧を求めない。
