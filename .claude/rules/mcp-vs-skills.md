## MCP vs Skills 判別ルール

新しい機能を追加する際、Skill で作るか MCP で作るかを判断する指針。
「対話ワークフロー = Skill、外部接続 = MCP」の使い分けが基本。

## 判別フロー

新機能の要件に対して、上から順に Yes/No を判定する。

1. **外部サービスのAPI接続 or 認証トークン管理が必要か？**
   → **Yes: MCP**（例: 各種 Web API, GitHub API, Slack, Gmail）
2. **Claude Code 以外（Claude Desktop / 他エージェント）でも同じ機能を呼びたいか？**
   → **Yes: MCP**（再利用性が資産になる）
3. **永続的な状態（DB、キャッシュ、認証セッション）を保持する必要があるか？**
   → **Yes: MCP**
4. **ユーザーとの対話・深掘り質問・整形済み出力が中心か？**
   → **Yes: Skill**（例: /journal, /weekly-review）
5. **複数ファイル・複数スキルにまたがるコンテキスト読み込み＋整形が中心か？**
   → **Yes: Skill**
6. **プロンプトの試行錯誤・微調整を頻繁にしたいか？**
   → **Yes: Skill**（Markdown 編集が速い。MCP はコード変更＋再起動が必要）

上記を通っても決まらない場合は **Skill から始める**（捨てやすく育てやすい）。

## Skill が向くケース

- 対話型ワークフロー（日記、自己分析、週次レビュー）
- コンテキスト整形・レビュー
- テンプレート駆動の文書生成
- プロンプトエンジニアリング主体・頻繁に編集したい機能

**特徴:** Markdown 1ファイルで完結し編集→即反映。対話履歴や他ファイルを自由に参照できる。状態は `context.yaml` などに逃がす。

## MCP が向くケース

- 外部サービス連携（各種 API）
- OAuth2 / APIトークン管理が必要なもの
- 構造化データ（JSON Schema で型付きの入出力を渡したい）
- 他のエージェント・ツールからも呼びたい共通ツール
- 長時間実行・ストリーミング・WebSocket 接続

**特徴:** 認証状態を永続化でき、型安全な `tool` インターフェースを持ち、複数クライアントから共有できる。実装コストは Skill より高い。

## 混在パターン: Skill が MCP を呼ぶ

最もパワフルな構成。Skill が対話ワークフローを担い、データ取得は MCP に委譲する。

- 対話が薄く、データ取得・操作が主体 → MCP 単独で十分
- 対話で深掘り・整形したい → Skill + MCP の2層

## 禁止事項

- Skill と MCP で同じ機能を二重実装しない。
- 実験・使い捨ての機能を MCP で作らない（Skill で試作 → 定着したら MCP 化を検討）。
- MCP を入れすぎない（ツール数の爆発はコンテキストを圧迫する）。**迷ったら Skill は作る、MCP は作らない。**
